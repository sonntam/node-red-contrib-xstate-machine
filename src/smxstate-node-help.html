Provides a runtime environment for state machines using XSTATE.

Properties

namestring
The name of the node as displayed in the editor
FSMjson
The json object which descibes the state machine. It needs to contain the initial state and possible transitions. For more information see details below
Send initial stateboolean
Enable to trigger sending the intital state after 100ms after the flow is started.
Show transition errorsboolean
Enable / disable error msgs when triggering actions.

Inputs

topic string
- reset to reset to initial state
- [transition] to trigger a state change
payload json
May contain a json payload, to add to the state output Example: { "x" : 10.0, y: "test" }

Outputs

    All outputs output the same data format: a msg, with its payload containing a status and a data object

  1. changed
    payload json
    Outputs the new state when there is any change in state.
  2. statusChanged
    payload json
    Outputs the new state only when the machine transitioned to a new status.
  3. dataChanged
    payload json
    Outputs the new state only when the data object of the state is changed.

Details

Example json code for FSM: 

{
  "state": {
    "status": "IDLE",
    "data" : { "x": 5 }
  },
  "transitions": {
    "IDLE": {
      "run": "RUNNING"
    },
    "RUNNING": {
      "stop": "IDLE"
    }
  }
}
  • state holds the initial state. It needs to contain a status field and might contain a data object
  • transitions holds the possible states as Keys (the upper case strings). As Values it holds one or more Key-Value Pairs, consisting of the transition (lower case strings) and the resulting state.
  • sending a msg with the topic set to the transition to the node, will trigger a state change.
  • reset is a reserved transition to reset the machine to its initial state, so it cannot be used in the transition table.
